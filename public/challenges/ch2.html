📖 Vulnerability Brief

SQL Injection occurs when user input is embedded directly into a database query without sanitization. By crafting special input, an attacker can break out of the intended query and execute their own SQL — reading data they should never see, bypassing logic, or even modifying the database. The Oracle shows you the query it ran. Use that to understand what is happening.

⚡ The Gate Oracle

Ask the Oracle when a gate opens by name. There are multiple gates. The Oracle reveals the query it used — study it carefully. Your goal: extract data the Oracle was not meant to show you. Three flags are hidden across the gates database.

💡 Hints

💡 Start by asking for a gate that exists — try Main Gate. Then look at the query_used field in the response. That is the actual SQL query the server ran with your input inside it.
💡 Your input goes inside single quotes in the query. What happens if your input also contains a single quote '? Try sending just ' as your input and read the error message carefully.
💡 To dump all rows, you need to make the WHERE clause always true. In SQL, OR can be used to add a condition that is always true. Think about what value is always equal to itself...
💡 The gates table has a hidden flag column not shown in normal queries. To read extra columns, look into the SQL UNION keyword — it lets you append results from a second SELECT statement. The UNION must return the same number of columns as the original query.

🔓 Solution & The Fix

Payloads that work + secure version
-- Dump all rows (WHERE always true)
' OR '1'='1

-- Extract the hidden flag column using UNION
' UNION SELECT flag, open_time FROM gates WHERE '1'='1

-- The vulnerability
const query = `SELECT name, open_time FROM gates WHERE name = '${name}'`;

-- The fix — parameterized query
const query = 'SELECT name, open_time FROM gates WHERE name = ?';
db.prepare(query).all(name);
